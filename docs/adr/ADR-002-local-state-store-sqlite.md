# ADR-002 本地运行态元数据持久化策略（SQLite）

- 状态: Proposed
- 日期: 2026-03-25
- 决策类型: 工程基础设施 / 状态持久化 / 审计

## 背景

当前项目通过文本日志和 `sync_state.json` 保存少量状态信息。

这种方式在项目早期可用，但随着工程化改造推进，已无法满足以下需求：

1. 记录每次同步作业的完整生命周期
2. 支持增量同步的稳定状态计算
3. 记录对象级审计信息并支持查询
4. 为错误恢复、重试、回归对比提供结构化数据
5. 支持版本化迁移和数据保留策略

同时，本项目当前仍是单机桌面应用，不引入独立服务端数据库会更符合部署复杂度和多数用户的接受度。

## 决策

引入本地 SQLite 作为“运行态元数据存储”，替代当前零散的 JSON 状态文件。

SQLite 的职责限定为：

1. 存储同步作业元数据
2. 存储对象级同步状态摘要
3. 存储审计与错误事件
4. 存储应用内部 schema version 与迁移状态
5. 支持本地查询、排障和增量同步

SQLite 不承担以下职责：

1. 不作为企业微信或 AD 的业务真相源
2. 不存储密码、Secret、Webhook 等敏感凭据
3. 不替代日志文件和最终审计输出

## 为什么选 SQLite

### 采纳原因

1. 单文件部署，适合当前桌面工具形态
2. 无需额外安装数据库服务
3. 支持事务，优于 JSON 文件的脆弱写入
4. 查询能力强，便于排障和对账
5. 支持 schema migration，利于长期演进

### 不直接继续用 JSON 的原因

1. 结构扩展困难
2. 并发写入和异常恢复能力弱
3. 缺少查询能力
4. 不利于对象级审计和增量同步

## 存储边界

### 允许存储的数据

1. `job_id`
2. 同步开始/结束时间
3. 作业状态
4. 统计信息
5. 对象标识
6. 对象摘要哈希
7. 目标 DN
8. 最后一次动作结果
9. 错误码、错误消息摘要
10. 配置 schema version

### 禁止存储的数据

1. LDAP 管理员密码
2. 企业微信 Secret
3. Webhook 完整地址
4. 明文默认密码
5. 完整 access token
6. 任何可直接用于身份认证的敏感凭据

## 建议的数据表

### 1. sync_jobs

记录同步作业级别信息。

建议字段：

- `job_id`
- `trigger_type`，例如 `manual` / `schedule`
- `status`
- `started_at`
- `ended_at`
- `total_users`
- `processed_users`
- `error_count`
- `summary_json`
- `app_version`

### 2. sync_events

记录作业过程中的关键事件。

建议字段：

- `id`
- `job_id`
- `stage`
- `level`
- `event_type`
- `message`
- `created_at`

### 3. object_sync_state

记录对象级增量同步摘要。

建议字段：

- `source_type`，例如 `wecom`
- `object_type`，例如 `department` / `user` / `group`
- `source_id`
- `source_hash`
- `target_dn`
- `last_seen_at`
- `last_action`
- `last_status`
- `extra_json`

### 4. object_operations

记录对象级变更动作，用于审计和追踪。

建议字段：

- `id`
- `job_id`
- `object_type`
- `source_id`
- `display_name`
- `action`
- `result`
- `target_dn`
- `error_code`
- `error_message`
- `created_at`

### 5. schema_migrations

记录 SQLite 自身的 schema 版本迁移。

建议字段：

- `version`
- `applied_at`
- `description`

## 配套设计原则

### 1. SQLite 是运行态数据仓，不是配置中心

运行参数仍然来自配置层，SQLite 只保存运行结果和状态摘要。

### 2. SQLite 是本地优化层，不是业务真相

企业微信部门树和 AD 当前状态仍然以外部系统为准。

### 3. SQLite 必须可重建

即使本地库损坏，系统也应能通过一次全量同步重建状态。

### 4. SQLite 需要保留策略

默认应支持：

1. 作业记录保留 N 天
2. 详细事件保留 N 天
3. 对象状态长期保留，但支持重建

## 备选方案

### 方案 A: 继续使用 JSON 文件

不采纳。

原因：

1. 无法支撑工程化状态管理
2. 查询和迁移能力不足
3. 易损坏且难排障

### 方案 B: 引入 MySQL / PostgreSQL

当前阶段不采纳。

原因：

1. 对单机桌面工具而言部署成本过高
2. 超出当前工程化阶段目标
3. 会显著提高多数用户的使用门槛

### 方案 C: 完全不做本地状态持久化

不采纳。

原因：

1. 无法稳定实现增量同步
2. 无法形成对象级审计
3. 难以支撑回归分析和问题定位

## 工程收益

1. 可支持真正的增量同步
2. 可查询单次同步处理了哪些对象
3. 可为 UI 提供历史作业视图
4. 可输出结构化审计报表
5. 可支持配置迁移和状态恢复

## 风险

1. 若将 SQLite 滥用为配置库，会带来新的安全问题
2. 若缺少 migration 机制，后续升级容易损坏状态
3. 若不设置清理策略，数据库会无限增长

## 实施要求

1. SQLite 文件路径独立于源码目录，放到应用数据目录
2. 启动时自动执行 schema migration
3. 所有写入通过仓储层封装，禁止业务逻辑直接拼 SQL
4. 敏感字段进入库前必须脱敏或禁止落库
5. 提供数据库修复、重建和清理命令

## 验收标准

1. 作业执行后可查询完整作业记录
2. 对象级变更可按 `job_id` 追踪
3. 增量同步状态不再依赖 `sync_state.json`
4. 删除本地 SQLite 后，系统仍可通过全量同步恢复
5. 仓库和打包产物中不包含敏感运行态数据库文件

## 结论

从工程化角度，当前项目应该引入本地 SQLite，但必须严格限定其边界：

1. 用来存运行态元数据和审计摘要
2. 不存凭据
3. 不替代外部系统真相源
4. 可迁移、可重建、可清理
