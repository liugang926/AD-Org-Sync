# ADR-006 同步执行模型与 Dry-Run 规则

- 状态: Proposed
- 日期: 2026-03-25
- 决策类型: 作业编排 / 执行安全 / 可观测性

## 背景

当前系统已经支持手工执行同步和定时执行同步，但执行模型仍然偏脚本式，存在以下问题：

1. 手工执行与定时执行没有统一作业模型
2. 缺少清晰的同步状态机
3. 缺少“仅预览、不落地”的安全模式
4. 缺少统一的作业编号、阶段状态和对象级变更计划
5. 一旦执行就直接修改 AD，首次上线和规则变更风险较高

从大多数用户视角，他们更需要的是：

1. 先看本次会改什么
2. 再决定要不要真正执行
3. 执行过程有明确状态
4. 同一时刻不要出现多个同步互相打架
5. 出问题后能定位是哪一次作业、哪一个对象、哪一个阶段失败

因此，需要统一同步执行模型，并明确 `dry-run` 与正式执行的边界。

## 决策

系统采用统一的“同步作业模型”，所有触发方式都进入同一套作业引擎。

同时引入两种执行模式：

1. `dry_run`
2. `apply`

其中：

1. `dry_run` 负责生成变更计划和风险预览，不对 AD 和企业微信目标状态做持久化修改
2. `apply` 负责按既定规则执行实际变更

## 统一作业入口

以下触发方式都必须进入统一作业引擎：

1. GUI 手工触发
2. 定时任务触发
3. 命令行触发
4. 后续可能增加的外部调度触发

区别仅体现在：

1. `trigger_type`
2. `execution_mode`
3. 调用方的确认流程

## 单作业原则

同一实例同一时刻只允许存在一个处于活动状态的同步作业。

活动状态包括：

1. `PLANNING`
2. `READY`
3. `RUNNING`
4. `CANCELING`

原因：

1. 防止并发修改 AD
2. 防止日志与状态相互污染
3. 简化排障
4. 降低用户认知复杂度

若已有活动作业，则新的触发请求应：

1. GUI 模式下明确提示“已有作业运行中”
2. 定时模式下跳过并记录原因
3. CLI 模式下返回明确退出码

## 状态机

### 1. Dry-Run 作业状态机

`CREATED -> PLANNING -> READY -> COMPLETED`

异常路径：

- `PLANNING -> FAILED`
- `PLANNING -> CANCELED`

### 2. Apply 作业状态机

`CREATED -> PLANNING -> READY -> RUNNING -> COMPLETED`

异常路径：

- `PLANNING -> FAILED`
- `RUNNING -> FAILED`
- `RUNNING -> CANCELING -> CANCELED`

说明：

1. `PLANNING` 阶段负责采集源数据、读取目标状态、计算差异和生成执行计划
2. `READY` 表示计划已生成且可执行
3. `RUNNING` 表示开始落地修改
4. `CANCELING` 表示已收到取消请求，等待到达安全取消点

## 规划与执行分离

同步过程分为两个大阶段：

1. `plan`
2. `apply`

### `plan` 阶段负责

1. 读取配置
2. 校验连接
3. 拉取企业微信部门与用户
4. 读取 AD 当前状态
5. 生成对象级差异
6. 生成变更计划
7. 计算风险与摘要

### `apply` 阶段负责

1. 执行 OU 创建
2. 执行组创建与组嵌套
3. 执行用户创建、更新、移动
4. 执行用户直属归组
5. 执行禁用/移入禁用 OU
6. 输出执行结果与审计数据

关键约束：

1. `dry_run` 只执行 `plan`
2. `apply` 必须先完成 `plan`，再进入 `apply`

## Dry-Run 规则

### 1. Dry-Run 的目标

为用户提供“本次会发生什么”的可信预览，而不是简单模拟日志。

### 2. Dry-Run 必须输出

至少输出以下内容：

1. 将创建的 OU
2. 将创建的组
3. 将建立或移除的组嵌套
4. 将创建、更新、移动、禁用的用户
5. 将新增或移除的直属组成员关系
6. 命中的保护/排除规则
7. 预计风险摘要

### 3. Dry-Run 不得执行

1. 不修改 AD 对象
2. 不创建或变更组嵌套
3. 不修改用户属性
4. 不写入外部系统状态
5. 不发送“成功执行”类通知

### 4. Dry-Run 可以写入

1. 本地作业记录
2. 本地审计摘要
3. 本地计划快照
4. 本地错误事件

## Apply 规则

### 1. Apply 必须基于计划执行

正式执行不得跳过 `plan` 阶段直接落地。

### 2. Apply 执行边界

仅执行当前计划中被判定为“需要应用”的对象级动作。

### 3. Apply 与规则漂移

若 `plan` 与 `apply` 之间配置发生变化，则：

1. 默认废弃旧计划
2. 重新生成计划

避免“按旧计划、用新规则”执行。

## 定时任务规则

从安全角度，定时任务默认执行模式应为：

- `apply`

但建议增加以下保护：

1. 首次上线前要求管理员先至少完成一次手工 `dry_run`
2. 大版本规则变更后，可配置为“下次计划任务先做 `dry_run` 并告警，不自动 apply”

建议配置：

- `schedule_execution_mode`
  - `apply`
  - `dry_run_then_notify`

默认建议：

- `apply`

## Feature Flag 建议

以下高风险能力应支持与作业模型联动的 feature flag：

1. `group_recursive_enabled`
2. `managed_relation_cleanup_enabled`
3. `user_disable_enabled`
4. `strict_ldaps_validation_enabled`

在 `dry_run` 中，这些 feature flag 仍应参与规划计算，以便用户看到真实预期结果。

## 审批与确认建议

从大多数用户视角，建议引入分层确认策略：

1. 普通手工同步
   - 可直接执行 `apply`
2. 首次启用递归组嵌套
   - 先 `dry_run`
3. 涉及批量禁用用户
   - 建议二次确认
4. 涉及受管关系清理
   - 强制 `dry_run`

## 对 SQLite 的要求

作业模型应与本地 SQLite 集成，至少持久化以下对象：

1. `sync_jobs`
2. `sync_job_stages`
3. `sync_events`
4. `planned_operations`
5. `executed_operations`

要求：

1. `dry_run` 和 `apply` 都必须有 `job_id`
2. `apply` 可引用其来源计划
3. 所有对象级动作都应可追踪到所属作业

## 错误处理规则

### 1. 规划期错误

若在 `plan` 阶段失败，则：

1. 作业状态置为 `FAILED`
2. 不进入 `apply`
3. 输出错误摘要

### 2. 执行期错误

若在 `apply` 阶段出现部分失败，则：

1. 保持对象级错误记录
2. 作业最终状态可为 `COMPLETED_WITH_ERRORS` 或 `FAILED`

建议新增终态：

- `COMPLETED_WITH_ERRORS`

原因：

1. 现实场景中常见部分成功
2. 便于区分“完全没执行”和“部分执行后失败”

## 取消规则

系统支持取消作业，但取消只能发生在安全边界。

建议安全取消点：

1. 部门同步阶段对象边界
2. 用户同步阶段对象边界
3. 组嵌套阶段对象边界
4. 禁用阶段对象边界

系统不得在单个 LDAP 修改操作中途强行中断。

## 可观测性要求

每次作业至少记录：

1. `job_id`
2. `trigger_type`
3. `execution_mode`
4. `status`
5. `started_at`
6. `ended_at`
7. `planned_operation_count`
8. `executed_operation_count`
9. `error_count`

每个阶段至少记录：

1. `stage_name`
2. `status`
3. `started_at`
4. `ended_at`
5. `object_count`
6. `error_count`

## 备选方案

### 方案 A: 继续脚本式直接执行

不采纳。

原因：

1. 风险高
2. 难以审计
3. 不利于工程化测试与回归

### 方案 B: Dry-Run 只打印推测日志

不采纳。

原因：

1. 不可信
2. 不能复用后续执行计划
3. 无法形成对象级审计

### 方案 C: Dry-Run 与 Apply 走两套完全独立逻辑

不采纳。

原因：

1. 极易产生行为漂移
2. 测试成本高
3. 维护成本高

## 风险

1. 如果计划阶段和执行阶段代码不共享同一套规则，将出现结果偏差
2. 如果不限制单作业，会出现并发冲突
3. 如果 `dry_run` 输出不完整，用户会误以为风险很低

## 验收标准

1. GUI、定时、CLI 触发都能生成统一作业记录
2. `dry_run` 不产生 AD 落地修改
3. `apply` 先规划后执行
4. 同一实例同一时刻不能运行两个活动作业
5. 作业、阶段、对象级动作都能通过 `job_id` 追踪
6. 高风险变更可通过 `dry_run` 预览

## 结论

系统采用“统一作业模型 + plan/apply 分离 + dry-run 预览”的执行机制。

这是将同步工具升级为工程化产品的关键基础设施之一。
