# ADR-005 受保护组与默认排除清单策略

- 状态: Proposed
- 日期: 2026-03-25
- 决策类型: 安全边界 / 权限保护 / 本地配置

## 背景

随着递归组层级和自动组同步能力增强，系统会开始处理更多 AD 组对象和组关系。

这会带来一个现实风险：

1. 域控中的大量默认内置组、特权组和系统组不应被自动同步逻辑误修改
2. 仅靠“组名是否看起来像默认组”来判断非常不可靠
3. 不同语言环境、重命名场景、不同林域结构下，默认组名称并不稳定

同时，用户也有真实诉求：

1. 希望系统默认不要动常见域控默认组
2. 希望这些组能体现在排除清单中，便于感知和管理
3. 希望自己能按业务需要增加或移除某些“排除组”

因此，系统必须同时满足“安全默认值”和“有限可配置性”。

## 决策

系统将 AD 组的保护与排除拆分为两层：

1. 硬保护组 `hard_protected_groups`
2. 默认排除组 `default_excluded_groups`

二者区别如下：

### 1. 硬保护组

特点：

1. 系统绝不自动修改
2. 系统绝不自动添加成员
3. 系统绝不自动建立组嵌套
4. 不允许用户在 UI 中解除保护

### 2. 默认排除组

特点：

1. 首次启动时自动写入排除清单
2. 默认不参与自动同步修改
3. 用户可以手工移除
4. 用户也可以手工新增其他排除组

这意味着：

1. “安全红线”由硬保护组保证
2. “业务灵活性”由默认排除组和用户自定义排除组保证

## 为什么不能让所有默认组都自由解锁

如果允许所有默认内置组都从排除清单中直接移除并参与同步，会带来高风险：

1. 误修改 `Domain Admins`、`Enterprise Admins` 等特权组
2. 误建立系统组嵌套，影响权限继承
3. 误加用户进入高权限组
4. 生产事故后几乎不可接受

因此，本 ADR 明确：

1. 可以让用户编辑“默认排除组”
2. 但不建议让用户解除“硬保护组”

## 组识别原则

系统识别受保护组和排除组时，不应只依赖组名。

优先级建议如下：

1. `objectSid`
2. 已知 RID / well-known SID
3. 组所在容器和对象属性
4. `sAMAccountName`
5. 显示名

原因：

1. 组名可能本地化
2. 组名可能被管理员重命名
3. 不同语言环境下名称不稳定

## 建议的硬保护组范围

硬保护组建议至少包含以下高风险默认组。

按 RID / well-known SID 识别，不按显示名硬编码：

1. `Domain Admins` RID 512
2. `Schema Admins` RID 518
3. `Enterprise Admins` RID 519
4. `Administrators` Builtin RID 544
5. `Account Operators` Builtin RID 548
6. `Server Operators` Builtin RID 549
7. `Backup Operators` Builtin RID 551
8. `Print Operators` Builtin RID 550

可选纳入硬保护的组：

1. `Domain Controllers` RID 516
2. `Read-only Domain Controllers` RID 521
3. `Protected Users` RID 525
4. `Key Admins` RID 526
5. `Enterprise Key Admins` RID 527

说明：

具体硬保护清单可以按客户安全策略调整，但默认应足够保守。

## 建议的默认排除组范围

默认排除组用于“默认不参与修改，但用户可调整”的场景。

建议包括但不限于：

1. `Domain Users` RID 513
2. `Domain Guests` RID 514
3. `Domain Computers` RID 515
4. `Users` Builtin RID 545
5. `Guests` Builtin RID 546
6. `Replicator` Builtin RID 552
7. `Group Policy Creator Owners` RID 520
8. 其他客户现场已约定不参与自动编排的基础组

## 用户可配置能力

系统应支持以下用户操作：

1. 查看系统识别出的硬保护组
2. 查看默认排除组
3. 新增自定义排除组
4. 移除默认排除组
5. 启用或禁用某条排除规则

系统不支持：

1. 解除硬保护组
2. 将硬保护组纳入自动同步修改范围

若未来必须支持“高级解锁模式”，应单独立 ADR，并至少满足：

1. 管理员二次确认
2. `dry-run`
3. 显式风险警告
4. 审计记录

## 本地持久化策略

这类策略应持久化到本地 SQLite。

原因：

1. 这是应用本地策略，不是外部系统真相
2. 需要 UI 可读写
3. 需要版本迁移
4. 需要记录来源和保护等级

## 建议的数据模型

```sql
CREATE TABLE group_exclusion_rules (
  id INTEGER PRIMARY KEY,
  rule_type TEXT NOT NULL,
  protection_level TEXT NOT NULL,
  match_type TEXT NOT NULL,
  match_value TEXT NOT NULL,
  display_name TEXT,
  is_enabled INTEGER NOT NULL DEFAULT 1,
  source TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

字段说明：

1. `rule_type`
   - `exclude`
   - `protect`
2. `protection_level`
   - `hard`
   - `soft`
3. `match_type`
   - `sid`
   - `rid`
   - `samaccountname`
   - `dn`
4. `source`
   - `system_seed`
   - `user_defined`
   - `migrated`

当前阶段推荐：

1. 硬保护规则主要按 `rid/sid` 存
2. 用户自定义排除规则优先按 `dn` 或 `sid` 存
3. 尽量不要只按显示名存规则

## 初始化策略

首次启动或升级到支持该规则的版本时：

1. 系统自动写入硬保护组规则
2. 系统自动写入默认排除组规则
3. 若本地已存在用户自定义规则，则保留不覆盖

## 同步执行时的处理顺序

系统处理任意组对象前，应按如下顺序判断：

1. 是否命中硬保护组
2. 是否命中排除规则
3. 是否属于系统管理组
4. 是否属于本次同步目标

判定结果：

1. 命中硬保护组 -> 直接跳过，不允许改动
2. 命中软排除组 -> 默认跳过
3. 未命中排除，且为系统管理组 -> 按同步规则执行
4. 非系统管理组 -> 默认不做结构性改动

## UI 与用户体验建议

建议在 UI 中分开展示：

1. 硬保护组
   - 只读
   - 不可移除
   - 明确标识“系统永不修改”
2. 默认排除组
   - 可启用/禁用
   - 可移除
3. 用户自定义排除组
   - 可新增
   - 可移除

建议增加以下交互：

1. 搜索组
2. 一键恢复默认排除清单
3. 风险提示
4. 预览哪些组将被跳过

## 审计要求

以下动作必须写入审计日志：

1. 用户新增排除组
2. 用户移除默认排除组
3. 用户启用或停用排除规则
4. 同步时因命中保护/排除规则而跳过某组

建议记录：

1. `job_id`
2. 规则来源
3. 规则命中类型
4. 组标识
5. 操作结果

## 与递归组层级规则的关系

本 ADR 对 ADR-004 做补充约束：

1. 即使组层级同步已启用，硬保护组仍不得参与自动嵌套
2. 默认排除组若仍在清单中，也不得被系统自动纳入父子组层级
3. 用户新增的排除组同样应从递归组层级处理中跳过

## 备选方案

### 方案 A: 所有默认组都允许完全自由编辑

不采纳。

原因：

1. 风险过高
2. 极易误碰高权限组
3. 不符合多数企业的最小惊扰原则

### 方案 B: 所有默认组都完全硬编码且不可见

不采纳。

原因：

1. 用户无法理解系统为什么跳过某些组
2. 不利于运维排障
3. 缺少灵活性

### 方案 C: 只靠组名做排除

不采纳。

原因：

1. 不可靠
2. 本地化和重命名会失效
3. 可能误伤业务组

## 风险

1. 若现场 AD 对默认组进行了重命名，仅按名称会识别失败
2. 若未区分硬保护和软排除，用户可能误解系统能力边界
3. 若 UI 不清晰，用户可能以为移除软排除就能修改硬保护组

## 验收标准

1. 常见域控特权组不会被自动修改
2. 默认排除组会自动写入本地排除清单
3. 用户可新增排除组
4. 用户可移除默认排除组
5. 用户不能解除硬保护组
6. 规则命中基于 SID/RID 等稳定标识，不依赖显示名
7. 同步日志中能明确看到“因保护/排除规则跳过”的原因

## 结论

从工程化与生产安全角度，系统不应仅使用一个“可任意编辑的排除清单”来保护域控默认组。

更稳妥的模型是：

1. 使用“硬保护组”守住安全底线
2. 使用“默认排除组”提供业务灵活性
3. 使用 SQLite 持久化规则与来源
4. 使用 SID/RID 作为识别基准，而不是依赖组名
