# ADR-007 默认数据库采用 SQLite 的部署与约束策略

- 状态: Proposed
- 日期: 2026-03-26
- 决策类型: 数据存储 / 部署策略 / 运维约束

## 背景

系统已经从单脚本工具演进为具备以下能力的工程化产品雏形：

1. 本地配置与规则管理
2. 同步作业模型与 `dry_run`
3. 审计记录与计划变更
4. 后续可能增加的 Web 控制台、登录、账户与自助改密

随着能力增加，系统需要一个稳定的关系型数据库来承载：

1. 配置元数据
2. 规则配置
3. 作业与审计数据
4. 账户与权限数据
5. 后续 Web 控制台运行态数据

当前存在一个明显的工程权衡：

1. `PostgreSQL` 更标准、更适合多实例和高并发
2. 但安装、运维、备份、发布链路对中小型内网场景偏重
3. 系统的实际业务负载并不属于高频写入或高并发场景
4. 当前产品目标优先级是“快速部署、低运维成本、单机稳定运行”

因此，需要明确默认数据库策略，而不是将数据库实现提前复杂化。

## 决策

系统默认数据库采用 `SQLite`。

同时明确以下部署与演进策略：

1. 默认部署模式为单机单实例
2. 默认执行模型为单进程服务 + 单 worker 串行任务
3. 默认数据库文件仅允许本机应用进程访问
4. 不以共享文件方式向多实例或多机器暴露数据库
5. 当系统超出既定边界时，再升级到 `SQL Server Express` 或 `PostgreSQL`

## 选择 SQLite 的原因

### 1. 符合当前真实负载

当前及中期规划中的主要负载包括：

1. 管理员登录
2. 配置读写
3. 规则维护
4. 定时或手工触发同步
5. 作业、事件与审计记录
6. 偶发的员工自助改密

这些负载具备以下特征：

1. 写入频率低
2. 并发量低
3. 更看重一致性与可部署性，而不是吞吐量
4. 大多数场景都发生在单台内网服务节点

### 2. 更符合快速部署目标

采用 SQLite 带来的直接收益包括：

1. 无需单独安装数据库服务
2. 无需初始化独立数据库实例
3. 备份与迁移成本低
4. 开源版和试用版启动门槛更低
5. 更适合 Windows 单机部署与内部工具场景

### 3. 当前不满足默认启用 PostgreSQL 的必要条件

除非出现以下需求，否则优先引入 PostgreSQL 的收益有限：

1. 多实例横向扩容
2. 持续高并发写入
3. 多 worker 并行作业
4. 高可用与主备架构
5. 独立 DBA 或标准化数据库运维体系

## 适用边界

本决策仅适用于以下默认场景：

1. 单台应用服务器
2. 单实例部署
3. 单个数据库文件
4. 单 worker 串行执行同步任务
5. 管理后台与自助页面共用同一应用服务
6. 内网访问，不对公网开放

## 不适用边界

一旦出现以下任一条件，应评估升级数据库方案：

1. 需要两个及以上应用实例同时写库
2. 需要多个 worker 并发执行任务
3. 审计、配置、会话写入增长明显，影响页面响应或任务稳定性
4. 数据库文件需要放置于共享盘、NAS 或 SMB 路径
5. 需要跨节点高可用
6. 需要对数据库执行复杂分析型查询
7. 需要接入企业已有数据库治理体系

## 升级路径决策

### 1. 首选升级路径：SQL Server Express

在以下场景下，优先考虑 `SQL Server Express`：

1. 部署环境以 Windows 和 AD 生态为主
2. 运维团队更熟悉微软栈
3. 系统仍然是单机或小规模单节点服务
4. 需要比 SQLite 更规范的数据库服务化能力

### 2. 高级升级路径：PostgreSQL

在以下场景下，优先考虑 `PostgreSQL`：

1. 计划多实例部署
2. 需要横向扩展
3. 需要更强的并发写入能力
4. 需要未来容器化、云化或高可用架构
5. 团队希望统一使用开源标准数据库栈

## 必须遵守的工程约束

默认使用 SQLite 时，必须满足以下约束：

1. 启用 `WAL`
2. 启用外键约束
3. 设置合理的 `busy_timeout`
4. 所有 schema 变更必须通过 migration
5. 所有核心表必须具备主键、必要唯一约束和索引
6. 数据库文件不得直接放在源码目录
7. 不允许将数据库文件部署在网络共享目录
8. 不允许多个应用实例同时写同一个数据库文件

## 安全边界

采用 SQLite 并不意味着可以弱化安全要求。

以下数据不得以明文形式存储在 SQLite 中：

1. LDAP 管理员密码
2. 企业微信 `CorpSecret`
3. 机器人 `Webhook`
4. 用户当前密码
5. 可复用的外部访问令牌

这些敏感数据应交由独立凭据机制处理，例如：

1. Windows Credential Manager
2. DPAPI
3. 后续可扩展的 secret provider

SQLite 中允许存储：

1. 密钥引用关系
2. 配置版本元数据
3. 角色与权限绑定
4. 审计记录
5. 作业与计划
6. 非敏感运行态状态

## 备份与恢复原则

默认部署必须支持以下能力：

1. 手工触发备份
2. 启动前自动快照
3. 计划性备份
4. 恢复后完整性检查
5. schema 版本校验

数据库备份目标不应只关注“文件复制成功”，还必须保证：

1. 可恢复
2. 可校验
3. 可追踪版本

## 对产品与架构的影响

本决策要求系统遵循以下架构原则：

1. 数据访问必须经过统一 repository / storage 层
2. 不允许将 SQL 直接散落在 UI 或控制器层
3. 数据库提供方必须可替换
4. 业务层不得依赖 SQLite 专有行为作为长期契约
5. Web 化时默认仍以单机单实例为主，避免过早复杂化

## 被拒绝的方案

### 1. 默认直接采用 PostgreSQL

拒绝原因：

1. 与当前真实负载不匹配
2. 显著增加部署和运维成本
3. 对开源用户和试用用户不够友好

### 2. 默认采用 SQL Server Express

拒绝原因：

1. 虽然对 Windows 友好，但仍需要独立数据库安装与初始化
2. 作为默认开箱方案仍不如 SQLite 轻量
3. 更适合作为企业环境升级选项，而非默认依赖

### 3. 继续沿用 JSON / INI / 零散文件组合

拒绝原因：

1. 一致性不足
2. 难以支撑 Web 化、多角色和审计
3. 迁移和升级能力弱

## 结论

在当前产品目标和常见部署场景下，`SQLite` 是默认数据库的最优解。

该决策建立在明确边界之上：

1. 单机
2. 单实例
3. 低并发
4. 强约束
5. 可升级

因此，后续工程化实现应以“默认 SQLite、具备升级路径”为原则推进，而不是提前为并不存在的高并发场景支付复杂度成本。
