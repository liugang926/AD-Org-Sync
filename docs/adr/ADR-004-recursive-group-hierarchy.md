# ADR-004 递归组层级规则

- 状态: Proposed
- 日期: 2026-03-25
- 决策类型: 组织模型 / 权限继承 / 同步规则

## 背景

当前系统已具备以下能力：

1. 可按企业微信部门树创建多层 OU
2. 可为每个部门创建对应 AD 组
3. 可将用户加入其所属部门对应的组

但当前系统仍存在以下缺口：

1. 组与组之间没有建立父子嵌套关系
2. 父部门无法通过组层级自然继承子部门成员
3. 用户与组的直接关系、组与组的继承关系没有明确边界
4. 若后续引入唯一组标识，仍需一套稳定的层级同步规则

从大多数用户视角，他们更容易理解以下模型：

1. 用户属于哪些部门，就进入哪些部门组
2. 子部门天然属于父部门
3. 查询父部门时能看到子部门成员
4. 手工加的特殊权限组不应被系统误删

因此，需要一套明确的递归组层级规则，指导后续实现、测试和迁移。

## 决策

系统采用“显式直属成员 + 递归父子组嵌套”的层级模型。

具体来说：

1. 用户直接加入其在企业微信中显式归属的部门组
2. 子部门组直接加入父部门组
3. 不为用户自动补写所有上级部门的直接成员关系
4. 父部门对下级成员的覆盖通过组嵌套获得，而不是通过批量直加用户获得

## 术语

### 1. 显式部门归属

企业微信用户资料中直接声明的部门列表。

### 2. 直属成员关系

用户直接加入某个部门组，或子部门组直接加入父部门组。

### 3. 继承成员关系

通过 AD 嵌套组产生的传递性成员关系，不在 AD 中额外写一层用户直加记录。

## 核心规则

### 规则一: 用户直属归组规则

用户直接加入其在企业微信中显式归属的每一个“有效部门组”。

说明：

1. 若用户显式属于 `华东/财务`，则直接加入 `华东/财务` 对应组
2. 若用户显式属于 `华南/财务` 和 `共享中心`，则直接加入这两个组
3. 若用户显式属于父部门本身，则直接加入父部门组

系统不得因为父子嵌套存在，就自动省略用户对显式部门组的直属关系。

原因：

1. 需要保留与源系统一致的语义
2. 便于审计“用户被哪个部门直接归属”
3. 避免未来源数据变化时无法区分显式与继承关系

### 规则二: 组直属嵌套规则

每个有效子部门组直接加入其“最近有效父部门组”。

注意：

1. 这里只建立“直接父子”嵌套
2. 不直接把子组一次性加入所有祖先组
3. 祖先覆盖依赖 AD 组嵌套的传递性

示例：

若部门树为：

- 集团
- 集团/华东
- 集团/华东/财务

则组嵌套关系为：

1. `华东/财务组 -> 华东组`
2. `华东组 -> 集团组`

而不是：

1. `华东/财务组 -> 华东组`
2. `华东/财务组 -> 集团组`
3. `华东组 -> 集团组`

原因：

1. 最小变更原则
2. 更易理解和排障
3. 更易做增量更新和差异比对

### 规则三: 继承只通过组嵌套获得

用户对上级部门的成员资格，默认只通过子组嵌套继承获得，不自动为所有祖先组追加直接用户成员关系。

原因：

1. 降低写入量
2. 降低变更面
3. 审计更清晰
4. 可避免“直属成员”和“继承成员”混淆

### 规则四: 用户主 OU 放置与组层级解耦

用户放在哪个 OU，与其加入哪些组是两个独立问题。

建议：

1. 主 OU 由主部门放置规则决定
2. 组归属由显式部门列表决定

因此：

1. 用户只落一个主 OU
2. 但可直接加入多个部门组

## 排除规则

### 1. 排除部门

若某部门被配置为排除部门，则该部门默认不参与以下动作：

1. 不创建对应 OU
2. 不创建对应系统管理组
3. 不建立其与父部门的组嵌套
4. 不将用户直接加入该部门组

### 2. 排除部门对子树的影响

当前阶段采用保守策略：

1. 若某部门被排除，则其整个子树默认不参与递归组层级构建

原因：

1. 规则简单
2. 风险低
3. 大多数用户更容易理解

后续若有需求，可另立 ADR 评估“跳过当前节点并压缩挂接到最近有效祖先”的策略。

## 系统管理边界

### 系统允许管理的关系

系统只管理以下两类关系：

1. 系统创建的用户到系统创建的部门组的直属成员关系
2. 系统创建的部门组到系统创建的父部门组的直属嵌套关系

### 系统不得主动删除的关系

系统不得主动删除以下关系：

1. 系统组与非系统组之间的手工嵌套关系
2. 非系统来源的特殊权限组关系
3. 管理员手工添加的非受管扩展权限关系

核心原则：

系统只能收敛自己负责的“受管关系”，不应误删手工权限模型。

## 受管关系识别

为支持安全收敛，系统应能识别“哪些组是系统创建并管理的组”。

建议依赖以下条件联合判断：

1. `sAMAccountName` 满足系统命名规则
2. 组对象带有系统写入的元数据
3. 可从本地状态库中解析到 `department_id -> group_dn` 映射

只有当子组和父组都属于“受管组”时，系统才允许自动修复或移除错误嵌套关系。

## 幂等与收敛规则

### 1. 幂等要求

重复执行同步，不应产生重复嵌套、重复报错或错误累积。

### 2. 收敛要求

若部门树发生变化，系统应在受管范围内将组嵌套关系收敛到当前期望状态。

例如：

1. 子部门从 `华东` 挪到 `华南`
2. 则系统应在受管组范围内：
   - 移除 `子组 -> 华东组`
   - 建立 `子组 -> 华南组`

### 3. 非受管关系保护

即便发生收敛，系统也不得移除：

1. 子组到非受管权限组的手工嵌套
2. 管理员人工建立的额外授权结构

## 多部门用户规则

若用户显式属于多个部门，则：

1. 用户直接加入每一个有效显式部门组
2. 上级继承继续通过嵌套关系获得
3. 不对多部门场景做“只保留叶子部门”的裁剪

原因：

1. 需要忠实反映源系统显式归属
2. 避免因为简化逻辑丢失业务语义

## 根部门规则

根部门是否创建对应组，按“部门是否参与同步”决定。

若根部门参与同步，则：

1. 根部门组可以存在
2. 一级部门组直接挂到根部门组之下

若根部门被排除，则：

1. 一级有效部门组不再向上建立系统管理的父组嵌套

## 与命名规则的关系

递归组层级不得依赖组名中的路径结构。

组层级应基于：

1. 企业微信部门树
2. `department_id`
3. `group_dn`

而不是基于：

1. 组名是否包含连接符
2. `displayName` 中的展示路径
3. 任何字符串拆分规则

## 可观测性要求

每次同步在处理组层级时，至少要记录以下信息：

1. `job_id`
2. `child_department_id`
3. `parent_department_id`
4. `child_group_dn`
5. `parent_group_dn`
6. 操作类型，例如 `group_nest_add` / `group_nest_remove` / `group_nest_skip`
7. 操作结果

## 实施建议

建议实现顺序如下：

1. 先按 ADR-001 建立部门唯一组标识
2. 构建完整部门树与有效部门树
3. 为所有有效部门解析并缓存 `group_dn`
4. 逐个建立“子组 -> 最近有效父组”的直属嵌套
5. 在受管组范围内执行差异收敛
6. 单独记录对象级审计事件

## Feature Flag 建议

首次上线递归组层级时，应通过配置开关灰度控制：

- `group_recursive_enabled`

上线建议：

1. 先 `dry-run`
2. 再对测试组织启用
3. 确认 AD 权限继承符合预期后再全量启用

## 备选方案

### 方案 A: 用户直接加入所有祖先组

不采纳。

原因：

1. 写入量大
2. 关系冗余
3. 难以区分直属与继承
4. 变更时更难收敛

### 方案 B: 只把用户加入叶子组，忽略显式父部门归属

不采纳。

原因：

1. 会丢失源系统显式部门归属信息
2. 对多部门场景不够准确

### 方案 C: 组层级完全不做递归

不采纳。

原因：

1. 父部门无法自然继承下级成员
2. 与多数用户对组织树权限的直觉不一致

## 风险

1. 若现网已有手工嵌套，需要在受管边界识别上足够谨慎
2. 若未先解决组唯一标识问题，递归组层级会放大同名组风险
3. 若不做 `dry-run`，首次启用可能影响现网权限继承

## 验收标准

1. 用户显式归属的每个有效部门，均存在对应直属组成员关系
2. 每个有效子部门组，都能正确加入最近有效父部门组
3. 父部门能通过组嵌套继承看到子部门成员
4. 同步重复运行不产生重复嵌套错误
5. 部门换父级后，受管嵌套关系能正确收敛
6. 系统不会误删非受管的手工权限关系

## 结论

系统采用“显式直属成员 + 递归父子组嵌套”的递归组层级模型。

用户直属归组与父级继承分开建模，系统只管理受管范围内的组关系，并通过 `dry-run` 与 feature flag 降低上线风险。
