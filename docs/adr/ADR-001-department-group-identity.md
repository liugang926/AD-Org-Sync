# ADR-001 部门唯一标识与 AD 组命名规则

- 状态: Proposed
- 日期: 2026-03-25
- 决策类型: 领域模型 / 身份建模 / AD 映射

## 背景

当前系统将企业微信部门映射为 AD OU，并为每个 OU 创建同名安全组。

现状存在三个核心问题：

1. 组唯一性依赖 OU 叶子名，默认假设“部门名全局唯一”。
2. 不同路径下 OU 可以重名，但 AD 组在当前实现中无法稳定区分。
3. 组查找依赖组名，而不是稳定唯一标识，容易误加到错误组。

示例：

- `集团/华东/财务`
- `集团/华南/财务`

这两个部门的叶子名都叫“财务”。如果继续使用叶子名建组，将出现冲突或错误归组。

同时，组名不允许依赖 `/` 这类路径分隔符，因此不能通过“完整路径组名”直接规避冲突。

## 决策

系统不再使用部门叶子名作为 AD 组的技术唯一标识。

改为采用“企业微信部门唯一 ID 驱动的组身份模型”：

1. 每个企业微信部门唯一映射到一个 AD 组实体。
2. AD 组的技术唯一标识使用企业微信 `department.id` 生成。
3. AD 组的可读名称与技术唯一标识分离。
4. 程序内部所有组查找、组嵌套、用户归组均按部门唯一 ID 或已解析的组 DN 进行，不再按叶子名模糊查询。

## 具体规则

### 1. 组唯一标识

- `external_department_id`: 企业微信 `department.id`
- `group_key`: `WECOM_D<department_id>`
- `sAMAccountName`: `WECOM_D<department_id>`

示例：

- 部门 ID `101` -> `sAMAccountName=WECOM_D101`
- 部门 ID `205` -> `sAMAccountName=WECOM_D205`

### 2. 组可读名称

由于组名不允许依赖 `/`，因此可读字段采用安全分隔方案：

- `cn`: `<部门叶子名>__D<department_id>`
- `displayName`: `<部门叶子名> [D<department_id>]`
- `description`: 保存完整路径与元数据

示例：

- `集团/华东/财务` -> `cn=财务__D101`
- `集团/华南/财务` -> `cn=财务__D205`

可选扩展：

- 若需要更强辨识度，可将 `displayName` 设为 `华东-财务 [D101]`
- 完整路径写入 `description`，例如：
  `source=wecom; dept_id=101; parent_id=12; path=集团/华东/财务`

注意：

1. 程序不得从 `cn` 或 `displayName` 反向解析层级关系。
2. 层级关系只由部门树和组嵌套关系决定。

### 3. 组对象位置

本期保持兼容性优先，组对象仍创建在对应部门 OU 下。

原因：

1. 与现有实现和运维认知更接近。
2. 便于现场管理员在对应 OU 下查看对象。
3. 可降低本期重构迁移成本。

后续若需要，可另立 ADR 评估“统一 Groups OU”模式。

### 4. 程序内部查找方式

程序内部不再使用以下方式：

1. 按部门叶子名查组
2. 按 `cn=部门名` 查组
3. 通过字符串规则从组名推导路径

程序内部应改为：

1. 先建立 `department_id -> group_dn` 映射
2. 用户加组时按 `department_id` 获取目标组 DN
3. 子组加入父组时按 `department_id` 获取父子组 DN
4. 若组已存在，则通过扩展属性或 `sAMAccountName=WECOM_D<id>` 精确命中

## 递归组层级配套规则

本 ADR 仅定义组身份，不单独定义组层级，但要求与后续递归组层级规则兼容：

1. 用户加入叶子部门组
2. 子部门组嵌套到父部门组
3. 父组不直接依赖名字识别子组

## 备选方案

### 方案 A: 继续使用部门叶子名作为组名

不采纳。

原因：

1. 无法处理重名部门
2. 无法保证程序精确查组
3. 后续递归组层级也会建立在错误基础之上

### 方案 B: 使用完整路径字符串作为组名

不采纳。

原因：

1. 路径包含 `/` 等不稳定字符
2. 名称长度不可控
3. 部门改名或换层级时迁移成本高

### 方案 C: 只保留可读名称，不存唯一技术标识

不采纳。

原因：

1. 程序无法可靠查找和更新对象
2. 不能支持工程化迁移和回归验证

## 影响

### 正面影响

1. 彻底解决同名部门导致的组冲突问题
2. 用户归组、组嵌套和审计都可精确追踪
3. 为增量同步、迁移脚本和回归测试提供稳定主键

### 负面影响

1. 现网若已存在旧命名组，需要设计迁移方案
2. 管理员需要适应“技术名”和“可读名”分离的模型

## 实施要求

1. 为 AD 组增加稳定元数据字段，至少记录 `dept_id` 和 `path`
2. 新代码禁止按部门叶子名查组
3. 所有测试场景必须覆盖“同名 OU 不同路径”
4. 为旧组命名规则提供兼容迁移脚本
5. 同步日志中要输出 `department_id`，不能只输出部门名

## 验收标准

1. `集团/华东/财务` 与 `集团/华南/财务` 可同时同步成功
2. 两者分别生成不同 `sAMAccountName`
3. 多部门用户不会被加到错误的同名组
4. 回归测试覆盖重名部门场景

## 待定问题

1. 组元数据字段具体落在哪个 AD 属性上
2. 旧组迁移是否采用“原地重命名”还是“新建并迁移成员”
3. 是否需要将组统一迁移到专门的 Groups OU
