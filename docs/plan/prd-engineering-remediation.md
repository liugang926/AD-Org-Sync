# Notting AD Sync 工程化整改 PRD

## 1. 文档信息

- 项目名称: Notting AD Sync
- 文档类型: 工程化整改 PRD
- 当前版本: v1.0
- 文档日期: 2026-03-25
- 适用范围: 企业微信到 AD/LDAPS 同步工具源码、桌面端、构建发布流程、运行配置和运维体系

## 2. 背景与问题定义

当前项目已经具备实际业务能力，但整体形态仍然是“单机可用脚本 + 打包产物”的交付模式，距离可持续维护、可审计、可测试、可发布的工程化项目还有明显差距。

当前主要问题如下：

1. 安全风险过高。
   `config.ini`、状态文件和日志中存在明文敏感信息，包括企业微信 Secret、Webhook、LDAP 管理员账号密码和默认密码。
2. 架构耦合严重。
   UI、配置、同步编排、LDAP 访问、企业微信 API、日志和报表全部耦合在少数大文件中，难以定位问题和扩展功能。
3. 文档与实现不一致。
   文档宣称部分配置可选、支持增量同步等能力，但代码未完整落地或校验逻辑相互冲突。
4. 组模型设计存在缺陷。
   目前以 OU 叶子名作为安全组标识，无法正确处理“不同层级 OU 同名”场景，也未建立父子组层级关系。
5. 测试体系缺失。
   现有 `test_ldap.py`、`test_wecom_api.py`、`quick_test.py` 更接近现场联调脚本，不具备自动化回归能力。
6. 构建与发布不规范。
   源码、`venv`、`dist`、`build`、日志和运行配置混放，缺少标准仓库结构、CI/CD 和版本管理。
7. 运行时稳定性不足。
   调度线程和同步线程模型较脆弱，缺少幂等控制、并发保护和清晰的任务状态管理。

## 3. 目标

### 3.1 业务目标

将项目从“可运行工具”升级为“可维护、可审计、可交付的工程化同步系统”，支撑后续持续演进。

### 3.2 产品目标

1. 保持现有核心同步能力不退化。
2. 支持多层 OU 与递归安全组层级。
3. 正确处理不同路径下 OU 重名导致的组命名冲突。
4. 支持安全、可配置、可审计的生产运行模式。
5. 形成可测试、可构建、可发布的标准工程体系。

### 3.3 工程目标

1. 完成模块化重构，移除单体脚本结构。
2. 建立配置分层、密钥管理、日志脱敏和权限边界。
3. 建立单元测试、集成测试、冒烟测试和打包验证流程。
4. 建立 Git 仓库规范、CI 流水线和版本发布流程。
5. 形成架构文档、运维手册和故障处理手册。

## 4. 非目标

1. 本期不做 AD 到企业微信的反向同步。
2. 本期不做多租户改造。
3. 本期不做 Web 化管理后台，仍以桌面端为主。
4. 本期不引入复杂分布式架构，优先完成单机工程化。

### 5.1 核心用户

1. IT 运维工程师
2. AD 管理员
3. 企业微信管理员
4. 项目维护开发人员

## 6. 现状评估

### 6.1 代码与目录现状

当前代码主要集中在以下文件：

- `wecom_sync_ui.py`: PyQt5 UI、托盘、调度、同步线程
- `wecom_sync_ad_ldaps.py`: 企业微信 API、LDAP/AD 同步、日志、报表、状态管理
- `wecom_sync_ad.py`: 历史版本实现
- `test_ldap.py`、`test_wecom_api.py`、`quick_test.py`: 联调脚本
- `*.spec`、`*.bat`: 打包与启动脚本

当前目录同时包含源码、构建产物、虚拟环境和运行数据，不符合标准工程布局。

### 6.2 安全现状

1. 敏感配置明文落盘。
2. LDAPS 证书校验关闭。
3. 日志存在敏感信息泄露风险。
4. 默认密码策略硬编码。
5. 运行账户权限边界不清晰。

### 6.3 模型设计现状

1. OU 可以按路径递归创建。
2. 每个 OU 会创建一个同名组。
3. 用户会加入其所在部门对应的组。
4. 组查找依赖叶子名，而不是唯一标识。
5. 不支持“子部门组加入父部门组”的递归层级。
6. 不支持“不同路径同名 OU”对应不同组的稳定映射。

## 7. 核心需求

### 7.1 需求一: 代码工程化重构

将现有单体结构拆分为清晰模块，建议目标结构如下：

```text
notting-ad-sync/
  src/
    app/
      bootstrap.py
      settings.py
    domain/
      models/
      services/
      policies/
    infrastructure/
      wecom/
      ldap/
      logging/
      storage/
    orchestration/
      sync_job.py
      scheduler.py
    presentation/
      desktop/
        main_window.py
        viewmodels/
    tests/
      unit/
      integration/
      smoke/
  scripts/
  docs/
  packaging/
  config/
    config.example.ini
  pyproject.toml
  README.md
```

要求：

1. UI 层不直接操作 LDAP 和企业微信 API。
2. 同步编排逻辑集中到 `orchestration`。
3. 领域规则集中到 `domain`。
4. 外部系统访问集中到 `infrastructure`。
5. 所有配置读取统一从 `settings` 层进入。

### 7.2 需求二: 安全配置治理

要求：

1. 仓库内不允许提交真实生产密钥。
2. 提供 `config.example.ini` 模板，不包含任何真实凭据。
3. 敏感信息优先从 Windows Credential Manager 或 DPAPI 读取。
4. 至少支持“配置文件 + 环境变量覆盖”两层机制。
5. 日志、报表、异常输出必须默认脱敏。
6. 启用 LDAPS 证书校验，支持证书路径或受信 CA 配置。
7. 默认密码不得硬编码为生产值。
8. 增加启动前安全自检，发现明文生产配置时提示风险。

### 7.3 需求三: 组织模型重构

#### 目标

解决多层 OU 与安全组模型不一致的问题，使 OU 树、组树和用户归属具备稳定、可推导、可回溯的一致性。

#### 规则

1. OU 路径按企业微信部门树递归创建。
2. 每个企业微信部门映射为一个唯一的 AD 组实体。
3. 组唯一标识不得使用 OU 叶子名，必须使用稳定唯一键。
4. 建议使用企业微信 `department.id` 作为组唯一标识来源。
5. 组显示名称可保留业务可读名，但技术标识必须唯一。

#### 组命名建议

建议采用双字段策略：

1. `sAMAccountName`: `WECOM_D<department_id>`
2. `cn/displayName`: `<部门完整路径>` 或 `<父路径>/<部门名>`

示例：

- 华东/财务 -> `sAMAccountName=WECOM_D101`, `displayName=华东/财务`
- 华南/财务 -> `sAMAccountName=WECOM_D205`, `displayName=华南/财务`

这样可以同时解决组名冲突和可读性问题。

### 7.4 需求四: 递归分组层级

要求：

1. 支持子部门组嵌套进入父部门组。
2. 用户默认加入所属叶子部门组。
3. 通过组嵌套实现父部门天然继承子部门成员。
4. 多部门用户可加入多个叶子部门组。
5. 用户的主部门 OU 放置规则要明确且稳定。

建议规则：

1. 用户所属 OU 使用企业微信主部门。
2. 若主部门不可用，则按部门 ID 排序后取最小值，避免“第一个返回结果”这种不稳定选择。
3. 用户加入所有所属叶子部门组。
4. 每个叶子部门组递归加入其所有上级部门组。

验收示例：

1. 用户属于 `集团/华东/财务`，则用户直接加入 `华东/财务` 对应叶子组。
2. `华东/财务` 组应嵌套进 `集团/华东` 组。
3. `集团/华东` 组应嵌套进 `集团` 组。
4. 查询 `集团` 组时能间接包含该用户。

### 7.5 需求五: 同步任务模型重构

要求：

1. 区分“计划任务调度”和“同步作业执行”两个职责。
2. 任何时刻只允许单个同步作业运行。
3. 同步作业具备明确状态机：
   `PENDING -> RUNNING -> SUCCESS / FAILED / CANCELED`
4. 支持任务取消，但取消点必须安全。
5. 对部门同步、用户同步、组同步、禁用同步分别记录阶段状态。
6. 增加幂等性保证，重复执行不会产生异常副作用。

### 7.6 需求六: 配置与文档对齐

要求：

1. 清理无效配置项。
2. 文档中所有配置项都必须有代码消费。
3. 所有代码消费的配置项都必须在文档中说明。
4. 对“必填/可选”做统一定义。
5. 将“默认值、推荐值、生产建议”写清楚。

### 7.7 需求七: 测试体系建设

测试分层建议如下：

1. 单元测试
   覆盖配置解析、OU 路径生成、组命名、用户筛选、状态计算、脱敏逻辑。
2. 集成测试
   使用 Mock 或测试替身验证企业微信 API 和 LDAP 客户端交互。
3. 合约测试
   验证企业微信返回数据结构变化对程序的影响。
4. 冒烟测试
   验证打包后的 EXE 能启动、能读取配置、能跑通最小同步流程。
5. 手工验证清单
   覆盖生产联调前的账号权限、证书、白名单、网络连通性检查。

最低验收要求：

1. 核心域逻辑单元测试覆盖率达到 70% 以上。
2. 关键同步链路至少具备 1 套集成测试。
3. 每次打包前自动运行单元测试与冒烟测试。

### 7.8 需求八: 构建发布与版本管理

要求：

1. 初始化 Git 仓库并建立分支策略。
2. 增加 `.gitignore`，排除 `venv/`、`dist/`、`build/`、`logs/`、真实配置和状态文件。
3. 使用 `pyproject.toml` 或标准打包入口统一依赖管理。
4. 标准化 PyInstaller 构建脚本。
5. 在 CI 中自动执行：
   - 代码检查
   - 单元测试
   - 打包验证
6. 每次发布产物都附带版本号、变更说明和校验信息。

### 7.9 需求九: 可观测性与审计

要求：

1. 统一日志格式。
2. 区分业务日志、审计日志、调试日志。
3. 敏感信息必须脱敏。
4. 每次同步生成唯一 `job_id`。
5. 所有部门、用户、组变更记录必须可追踪到 `job_id`。
6. 错误报告中应包含失败对象、失败动作、错误码和建议动作。
7. 审计文件建议输出结构化 JSON 和便于阅读的文本摘要两份。

### 7.10 需求十: 运维与交付能力

要求：

1. 提供部署手册。
2. 提供配置手册。
3. 提供联调手册。
4. 提供故障排查手册。
5. 提供升级回滚手册。
6. 提供生产上线检查表。

## 8. 关键架构建议

### 8.1 分层建议

推荐采用“桌面端壳层 + 应用服务 + 领域服务 + 适配器”的分层架构。

原则：

1. Presentation 只负责收集输入和展示状态。
2. Application/Orchestration 负责同步作业编排。
3. Domain 负责部门映射、用户规则、组规则、禁用策略。
4. Infrastructure 负责对接 LDAP、企业微信、文件系统和通知渠道。

### 8.2 领域模型建议

建议抽象以下模型：

1. DepartmentNode
   包含 `department_id`、`name`、`parent_id`、`full_path`
2. UserAccount
   包含 `userid`、`name`、`email`、`main_department_id`、`department_ids`
3. GroupBinding
   包含 `department_id`、`group_sam`、`group_dn`、`display_name`
4. SyncJob
   包含 `job_id`、`status`、`started_at`、`ended_at`、`stats`
5. SyncDecision
   表示某对象本次应 `create/update/disable/skip`

### 8.3 规则引擎建议

建议将以下规则显式化，不要散落在主流程中：

1. 用户归属 OU 规则
2. 组命名规则
3. 排除部门规则
4. 排除账号规则
5. 邮箱回填规则
6. 禁用而非删除规则
7. 递归组嵌套规则

### 8.4 状态管理建议

1. 用结构化状态文件记录上次成功同步时间、对象摘要、版本号。
2. 状态模型需带 schema version，便于升级兼容。
3. 真正启用增量同步前，先定义“变更判定字段集”。
4. 状态文件损坏时应自动回退并报警。

## 9. 改造方案

### Phase 0: 紧急止血

目标：先把安全和发布风险降下来。

任务：

1. 轮换当前所有暴露过的生产凭据。
2. 清理仓库中的真实敏感配置。
3. 新增 `.gitignore`。
4. 提供 `config.example.ini`。
5. 禁止日志打印完整 Secret、Webhook 和密码。
6. 关闭 README 和代码中的矛盾配置。

验收标准：

1. 仓库中不再包含真实凭据。
2. 运行日志中不再出现完整敏感字段。
3. 新开发环境可通过模板配置完成启动。

### Phase 1: 结构重构

目标：完成代码分层和目录重组。

任务：

1. 初始化 Git 仓库。
2. 引入 `src/` 结构。
3. 拆分 UI、客户端、同步编排、配置和日志。
4. 将历史脚本迁移到 `legacy/`。
5. 统一依赖声明到 `pyproject.toml`。

验收标准：

1. 主入口不再依赖单一超大文件。
2. UI 不直接调用 LDAP 原子操作。
3. 依赖安装和运行命令标准化。

### Phase 2: 组织模型重构

目标：修复 OU/组映射缺陷。

任务：

1. 建立部门唯一 ID 到组的映射模型。
2. 改造组创建逻辑，废弃“叶子名唯一”假设。
3. 增加子组加入父组逻辑。
4. 改造用户落 OU 规则为稳定规则。
5. 增加重名 OU 回归测试。

验收标准：

1. 同名 OU 不再导致组冲突。
2. 父组可继承子组成员。
3. 多部门用户归属稳定且可解释。

### Phase 3: 测试与流水线

目标：建立最小工程保障能力。

任务：

1. 引入 `pytest`。
2. 建立单元测试和集成测试基线。
3. 新增静态检查。
4. 增加 CI 脚本。
5. 新增打包冒烟验证。

验收标准：

1. PR 提交后自动运行校验。
2. 打包失败可在 CI 中直接暴露。
3. 同步核心逻辑具备自动回归能力。

### Phase 4: 发布与运维

目标：形成稳定交付闭环。

任务：

1. 发布版本规范化。
2. 补齐架构文档、运维手册、故障手册。
3. 增加升级与回滚策略。
4. 明确生产环境变更审批流程。

验收标准：

1. 可独立完成版本发布。
2. 版本变更可追溯。

## 11. 风险与依赖

### 风险

1. 旧配置与新配置格式不兼容。
2. 组命名规则变更可能影响现网权限模型。
3. 递归组嵌套可能影响现有 AD 权限继承。
4. 启用证书校验后，现网证书配置可能暴露问题。
5. 重构过程中若缺少测试，容易引入行为回归。

### 依赖

1. 企业微信部门 ID 稳定且可获取。
2. AD 管理员确认新组命名规则可接受。
3. 运维确认部署机支持安全凭据存储方案。
4. 团队接受 Git、CI 和版本管理流程。
