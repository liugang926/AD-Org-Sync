# SQLite 部署与运维工程规范清单

## 1. 目标

本规范用于约束系统在默认 `SQLite` 模式下的工程实现、部署方式与运维操作，确保：

1. 默认部署足够简单
2. 单机运行足够稳定
3. 数据一致性可接受
4. 后续可平滑升级到 `SQL Server Express` 或 `PostgreSQL`

## 2. 适用范围

本规范适用于以下场景：

1. 单机单实例 Web 服务
2. 桌面版与服务版共用本地数据库
3. 内网环境
4. 低并发、低写入频率

本规范不适用于：

1. 多实例共享数据库文件
2. 数据库文件部署在共享盘
3. 多 worker 并发写入
4. 高可用和多节点部署

## 3. 数据库存放位置

数据库文件不得放在源码目录、临时目录或网络共享目录。

建议默认位置：

- Windows: `%APPDATA%/ADOrgSync/app.db`

同目录建议包含：

1. `app.db`
2. `app.db-wal`
3. `app.db-shm`
4. `backups/`

## 4. 初始化要求

应用初始化数据库连接时，必须执行以下设置：

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA synchronous = NORMAL;
PRAGMA temp_store = MEMORY;
PRAGMA busy_timeout = 5000;
```

建议附加设置：

```sql
PRAGMA cache_size = -20000;
PRAGMA mmap_size = 268435456;
```

说明：

1. `WAL` 用于提升读写并存能力
2. `foreign_keys` 用于防止弱约束数据破坏
3. `busy_timeout` 用于降低短时锁冲突失败率
4. `synchronous` 可根据部署级别在 `NORMAL` 与 `FULL` 间切换

## 5. schema 设计要求

### 5.1 必须项

1. 每张核心表必须有主键
2. 具备业务唯一性的字段必须有唯一约束
3. 外键关系必须显式声明
4. 关键查询路径必须建立索引
5. 所有核心表必须包含 `created_at`
6. 可变更核心表必须包含 `updated_at`

### 5.2 建议项

1. 使用 `TEXT` 存储时间时统一采用 UTC ISO8601
2. 对枚举型字段使用明确字符串值
3. 对 JSON 字段限定用途，仅用于摘要和扩展信息
4. 避免把列表型主业务结构直接塞入 JSON

## 6. migration 要求

1. 所有 schema 变更必须通过 migration 执行
2. 应用启动时必须检查当前 schema 版本
3. 迁移必须具备幂等性
4. 迁移失败时应用不得继续进入写模式
5. 重大迁移前必须先创建数据库快照

最低要求：

1. `schema_migrations` 表
2. 当前版本号
3. 启动时自动升级
4. 升级日志

## 7. 写入模型约束

默认模式下必须遵守以下约束：

1. 同一时刻只允许一个同步作业执行
2. 应用内高风险批量写入必须串行化
3. 配置发布与正式同步不得并发落库
4. 密码相关操作必须走受控事务

推荐实现：

1. 应用级作业锁
2. 高风险操作状态机
3. 明确区分计划阶段与执行阶段

## 8. 备份与恢复要求

### 8.1 备份

必须支持以下备份方式：

1. 手工备份
2. 启动前快照
3. 定时备份

建议策略：

1. 每日增量或全量快照
2. 保留最近 `7` 份日备份
3. 保留最近 `4` 份周备份

备份文件命名建议：

- `app_YYYYMMDD_HHMMSS.db`

### 8.2 恢复

恢复流程必须包含：

1. 停止应用写入
2. 备份当前损坏或待回退数据库
3. 恢复目标快照
4. 启动完整性检查
5. 校验 schema 版本
6. 记录恢复操作审计

### 8.3 完整性检查

至少支持：

```sql
PRAGMA integrity_check;
```

## 9. 安全要求

以下内容不得明文存储于 SQLite：

1. LDAP 管理员密码
2. 企业微信 `CorpSecret`
3. Webhook
4. 用户当前密码
5. 可复用访问令牌

SQLite 中可以存储：

1. 凭据引用标识
2. 配置版本
3. 作业审计数据
4. 用户角色
5. 非敏感状态数据

本地数据库文件目录应满足：

1. 仅应用运行账户可写
2. 非管理员普通用户不可随意读取
3. 备份目录权限与主库一致

## 10. 观测与维护要求

应用必须至少暴露以下观测信息：

1. 当前数据库路径
2. schema 版本
3. 数据库文件大小
4. 最近备份时间
5. 最近一次完整性检查结果

建议增加：

1. 活跃作业数
2. 最近失败迁移记录
3. 最近锁等待告警

## 11. 升级触发条件

满足以下任一条件时，应评估升级数据库：

1. 单机写入锁争用频繁影响用户体验
2. 审计、会话、作业数据增长导致维护明显困难
3. 需要两个及以上应用实例
4. 需要多个 worker 并发执行
5. 需要高可用或主备切换
6. 企业要求统一纳管数据库服务

升级建议顺序：

1. Windows 企业环境优先评估 `SQL Server Express`
2. 多实例或长期平台化优先评估 `PostgreSQL`

## 12. 验收点

### 12.1 部署验收点

1. 应用首次启动可自动创建数据库
2. 默认配置下无需独立数据库安装
3. 数据库不落在源码目录
4. WAL 文件可正常生成

### 12.2 工程验收点

1. 应用启动时自动执行 PRAGMA 初始化
2. schema 迁移成功后才进入服务可写状态
3. 所有核心表具备主键和必要索引
4. 高风险任务默认串行

### 12.3 运维验收点

1. 可一键手工备份
2. 可从备份恢复
3. 恢复后可完成完整性检查
4. 备份和恢复操作有审计记录

## 13. 验收标准

1. 首次启动到可用数据库初始化完成时间不超过 `30` 秒
2. 启动后 `PRAGMA foreign_keys` 必须为 `ON`
3. 启动后 `journal_mode` 必须为 `WAL`
4. 所有 migration 执行结果必须可追踪
5. 同一实例下不得出现两个正式同步任务同时写库
6. 默认备份恢复演练成功率应达到 `100%`
7. 不允许在审计日志中出现明文密码、Secret、Webhook

## 14. 落地建议

第一阶段必须实现：

1. PRAGMA 初始化
2. migration 管理
3. 自动建库
4. 手工备份
5. 启动前快照
6. integrity check

第二阶段再实现：

1. 定时备份
2. 数据库健康页
3. 升级数据库提供方的迁移脚本
